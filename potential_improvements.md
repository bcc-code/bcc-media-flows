# Potential improvements

- ~~`services/transcode/preview.go::GrowingPreview` has the same `trc:reserved` swscaler-failure exposure that was just fixed in `Preview` and `VideoH264`.~~ Resolved by the growing-preview VU meter change: `GrowingPreview` now probes `input.FilePath` upfront and applies `NormalizeColorTRCFilter` (falls back to the legacy filter if the probe fails).
- `services/transcode/preview.go::GrowingPreview` calls `os.Exit(1)` if creating the tail→ffmpeg pipe fails. Library/activity code should return the error instead of killing the whole worker process.
- `workflows/ingest/incremental_ingest.go` (`doIncremental`): `previewTempPath.Append("preview")` discards its return value — `Append` returns a new path, so this line is a no-op and the preview temp files land directly in the workflow temp folder instead of a `preview` subfolder.
- `workflows/export/shorts.go::createShortInPlatform` still iterates `tagCodes` after the Notion→ClickUp migration, but `tagCodes` is now always empty (Source/Purpose/Type were dropped because ClickUp has no equivalents). Either re-introduce ClickUp custom fields for those tags (and wire them into `shortsDataFromClickUpTask`) or delete the loop entirely once it's clear we will not bring them back.
- `workflows/export/shorts.go::triggerShortExport` hardcodes `Languages: []string{"nor"}` after the Notion field was dropped. If shorts in other languages ever need to be exported, either add a `Language` custom field on the ClickUp "Shorts Export" list and reintroduce it on `ShortsData`, or derive it from the task name suffix (e.g. `..._NOR` → `nor`).
- `workflows/export/shorts.go::ShortsData` fields `InHm`/`OutHm` are named after `HH:MM:SS`, but ClickUp stores `MM:SS` (`convertToSeconds` already accepts both formats by prepending `"0"`). Consider renaming or commenting the expected formats.
- `workflows/vb_export/vb_export_test.go::TestVBExportTestSuite/Test_VBExportToXDCAM` fails on master as well as on this branch with `unable to decode: drive not found` — it appears to require `ISILON_PREFIX` to point at a real path. Worth fixing the test fixture or skipping when the prefix is unset so CI does not stay red.
- ffmpeg-based transcode functions in `services/transcode/` (e.g. `playout_mux.go`, `merge.go`, `h264.go`, `prores.go`, `hap.go`, `avc_intra.go`, `mux_mxf_simple.go`, audio/video variants) write an output path directly to `ffmpeg.Do` without first running `os.MkdirAll(filepath.Dir(out), ...)`. `Mux` in `mux.go` just hit this in production (missing `output/vod/` directory) and was fixed locally. The same latent bug likely affects the other call sites — worth a consistent sweep to mirror the convention used in `activities/files.go` (`MoveFile`, `CopyFile`, `WriteFile`).
- `workflows/vb_export/vb_export.go::VBExport` has dead/duplicated error handling in the dispatch loop and result collection: `if err != nil` right after `workflow.ExecuteChildWorkflow` can never fire (the future's error is only returned by `future.Get`), and in the result loop `errs = append(errs, err)` runs twice for the same error (once before and once inside a redundant nested `if err != nil`), so each child failure is reported twice in the aggregated error message. Also `var w interface{}` could be `var w any`.
- `services/vidispine/export.go::enrichClipWithEmbeddedAudio` mishandles the `zxx` (no linguistic content / music) language on the **16-channel** path (uses `MU1ChannelStart` = -1) and the **softron 64-channel** path (uses `SoftronStartCh` = -1). For `zxx`, `uint(l.MU1ChannelStart + i)` / `uint(langInfo.SoftronStartCh)` underflow to a garbage stream ID — no error, just wrong/garbage channels. The simpler embedded branches (0/1/2/8 components) handle `zxx` correctly. This is the *embedded* audio-source path; the related-audio path was just fixed to fall back to the clip's own original audio for `zxx`. Worth handling `zxx` (and any language with -1 channel offsets) explicitly on these two embedded branches too.
- `workflows/vb_export/vb_export.go::VBExport` (loop starting ~L226): after `future := workflow.ExecuteChildWorkflow(ctx, w, childParams)` the code checks `if err != nil` (L246), but `err` is the stale value from the earlier `CreateFolder` call — `ExecuteChildWorkflow` returns a future, not an error. Dead check; child-start issues surface only via `future.Get`. The vet `nilness` analyzer flags it as an impossible condition.
- `workflows/vb_export/vb_export.go::VBExport` (result-collection loop ~L260-266): inside `if err != nil { errs = append(errs, err); SendTelegramText(...); if err != nil { errs = append(errs, err) } }` the nested `if err != nil` is always true, so each failed child export error is appended to `errs` twice. Remove the redundant inner block.
